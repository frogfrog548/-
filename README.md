## б) Постановка задачи

Разработать консольное приложение «Электронный дневник», позволяющее учителю или классному руководителю вести учёт успеваемости и посещаемости учеников. Программа должна обеспечивать:

1. **Управление списком учеников** – добавление новых учеников с автоматической генерацией ID.  
2. **Управление предметами** – добавление учебных дисциплин.  
3. **Выставление оценок** – оценка по 5-балльной шкале (2–5) с привязкой к ученику, предмету и дате.  
4. **Просмотр успеваемости** – вывод всех оценок ученика по предметам, расчёт среднего балла.  
5. **Домашние задания** – добавление задания (предмет, описание, срок сдачи) и просмотр списка (всех или по предмету).  
6. **Посещаемость** – отметка присутствия/отсутствия всех учеников на текущую дату; просмотр статистики посещаемости конкретного ученика (общее количество, процент явки).  
7. **Постоянное хранение данных** – все данные сохраняются в файлах и восстанавливаются при следующем запуске.  
8. **Консольный интерфейс** – простое текстовое меню с выбором действия по номеру.

## в) Анализ задачи и структура классов

### 1. Общая архитектура

Проект реализован в объектно-ориентированном стиле. Выделены следующие логические слои:

- **Модели данных** – классы, описывающие предметную область (`Student`, `Subject`, `Grade`, `Homework`, `Attendance`).  
- **Слой хранения** – класс `DataManager`, инкапсулирующий работу с файлами (сериализация/десериализация в JSON).  
- **Прикладной слой** – класс `ElectronicDiary`, содержащий бизнес-логику и консольный пользовательский интерфейс.

### 3. ООП и принципы

#### Инкапсуляция
- Данные каждого класса хранятся в атрибутах. Внутри `ElectronicDiary` все вспомогательные методы названы с подчёркивания (`_next_id`, `_show_students`, `_get_int`), что сигнализирует об их внутреннем использовании.  
- `DataManager` полностью скрывает детали файловых операций: методы `_path`, `_load`, `_save` являются приватными, а публичный интерфейс предоставляет только специализированные методы (`load_students`, `save_students` и т.д.). Внешний код не знает, какие имена файлов используются и как происходит сериализация.  
- Класс `ElectronicDiary` не позволяет напрямую манипулировать списками данных – все изменения производятся через методы приложения, которые заодно сохраняют изменения в файлы.

#### Наследование
В данной предметной области не используется, так как сущности (`Student`, `Subject`, `Grade` и т.д.) не имеют общего поведения, кроме методов `to_dict`/`from_dict`. Создавать абстрактный базовый класс было бы излишним усложнением («вымучиванием»), что противоречит критерию. Поэтому от наследования разработчик обоснованно отказался.

#### Полиморфизм
Не требуется. Однако принцип подстановки Лисков не нарушен – отсутствие иерархии не мешает работе.

### 4. Хранение данных

**Формат JSON** выбран по следующим причинам:
- Человекочитаемость – при необходимости можно открыть и отредактировать файл вручную.  
- Лёгкая сериализация/десериализация (стандартная библиотека `json`).  
- Хорошо подходит для хранения списков объектов с плоской структурой.  
- Каждая сущность хранится в отдельном файле:
  - `students.json`
  - `subjects.json`
  - `grades.json`
  - `homeworks.json`
  - `attendance.json`

Класс `DataManager` обеспечивает единообразное чтение и запись: метод `_load` получает имя файла и класс модели, читает JSON, преобразует каждый элемент в объект через `cls.from_dict`. Аналогично `_save` преобразует список объектов в словари и сохраняет.

### 5. Пользовательский интерфейс

Консольное меню с числовым выбором действий (1–8, 0 – выход). Каждое действие вызывает соответствующий метод класса `ElectronicDiary`.  
- Интерфейс содержит подсказки для ввода (`ID ученика:`, `Оценка (2-5):` и т.д.).  
- Перед запросом ID выводятся списки учеников и предметов для наглядности.  
- Программа устойчива к некорректному вводу (при вводе букв вместо чисел метод `_get_int` возвращает `None` и выводит сообщение об ошибке).
